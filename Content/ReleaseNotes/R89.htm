Salesforce: New parameters to segment work queue updates

New parameters were added to the Salesforce API for additional customizations:

WorkQueueRecordsToProcess: Defines the max number of records to be processed at one time (defaults to 2000).

IgnoreWorkQueueErrors: Allows the admin to ignore work queue failures and mark them as completed so they are not reprocessed.

Admin must have the ability to add an available custom field to the work queue record in Salesforce that is available for read/write through the API. Can be a rich text field to preserve log formatting.

41964

Tracking the source of data change in Prism: Additional Record Information for Dataform Imports

The PrismHR import API now includes a column for Additional Information. This data is available to all dataforms at the time of import and contains information about how the related record was changed in PrismHR. Currently, the fields available are prismUserName and userType. The Additional Record Information virtual field displays as available trigger information in Email Template and Workflow Channel configuration, enabling the system administrator to perform automatic notifications and workflow based on the prismUserName and userType responsible for a PrismHR change. Additional Record Information data is only available as a trigger condition from the Triggering Form configured on a Workflow Channel or Email Template. For more information, see Tracking Additional Record Information in Prism events: Triggering from Additional Information.

42151

 

Storing old Email Address values

The Employee dataform now includes Previous Work Email and Previous Personal Email. When an Employee’s Work Email and Personal Email changes occur in PrismHR, a business rule sets the Previous Email fields. This enhancement enables configuring an Email Template or Workflow Channel to alert the employee that this information changed.

The enhancement includes:

New Previous Work Email and Previous Personal Email fields on the Employee dataform (read only).

A new business rule (SetPreviousEmails) on the Employee dataform for when PersonalEmail or WorkEmail is changed. This rule stamps the old value into the corresponding PersonalEmailPrevious and WorkEmailPrevious fields. This rule is set to run on the Default and Imports pipelines.

Previous Personal Email and Previous Work Email are both valid options as recipients for an Email Template triggered by the Employee dataform. They display and send a notification to the stored, old, email address values.

The peo_directdeposit_emailtemplate_recipient_v2_proc was enhanced to include the Previous Personal Email and Previous Work Email.

For additional information, see Configuring notifications for Employee email address changes.

42159

 

Remove Dataform locking logic for missing header

Dataforms were updated to remove the locking logic, which made the form read-only if the form configuration contained an unsupported header configuration in our cross-browser platform NEXT. Previous to this development, a red banner displayed at the top of a dataform with an invalid header “This form is not supported at this time in NEXT.” Form fields are now secured purely based on dataform security as with any other dataform in the system.
42214

PrismHR Import enhancements

The PrismHR Import system was enhanced with a new PrismHR Import Initial Locations scheduled process, scripted as active, with the next run date set. This scheduled process uses the HRPAPI_initial_location*.csv import configuration (must be active) to retrieve a list of Active ClientMaster records where Client Number is not empty. These Client Numbers are then matched in PrismHR to create a list of locations for each client. See List of scheduled processes.

If the client number cannot be matched in PrismHR, the client is ignored. These locations are then mapped into an import file for all clients based on the initial_location import configuration. When all locations for all clients are retrieved, the import begins. This process seeds the system with any missing or new client locations, at the end of which the scheduled process inactivates itself. See PrismHR Import Queue initial clients scheduled process.

43161

Payroll Batch Imports

Payroll Batch Ongoing Imports were enhanced to handle failed or deleted payroll batches in PrismHR.

Payroll Batch DELETE events are processed to create Import Queue records.

When DELETE events are processed to generate Import file rows, they are marked Completed with the message: "Batch '#####.###' Deleted, this record will not be processed"

If we are unable to retrieve a Payroll Batch from an event, that event (Import Queue record) is marked as Completed with a message: "Batch '#####.###' Unable to include Queue Record in Import file, no getBillingCodeTotalsForBatch response info, this event will not be processed"

See Payroll Batch information.
